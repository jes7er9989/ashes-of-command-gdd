L.4.2 Reserve Wave System
RESERVE DEPLOYMENT
Reserves defined per-team in battle config
Each wave: unit_id, count, deploy_delay (seconds)
Spawns at team back edge (TEAM_OFFSET_X + 6.0m)
1.0s spawn protection (immune + fade-in tween)
HUD notification with 2s display + 1s fade
LAST-STAND MECHANIC
If all field units die but reserves remain:
→ Force-deploy ALL pending reserves immediately
Prevents instant-loss when reinforcements are queued
Victory deferred until force-deploy resolves
Outcomes: PLAYER WINS / ENEMY WINS / DRAW
L.5 Battle Pipeline — 11-Step Combat Formula
#OPERATIONFORMULA
1Accuracy Rollrand() < attacker.accuracy → hit or miss
2Base Damageattacker.attack_damage
3Critical Hitrand() < crit_chance → dmg × crit_multiplier
4Evasion Rollrand() < target.evasion → attack dodged
5Resistancedmg × (1.0 − target.resistances[dmg_type])
6Armor Reductiondmg = max(1, dmg − target.armor)
7Damage Floormax(1, final_dmg) — always at least 1
8Shield Absorbshield absorbs first; overflow → HP
9HP Deductiontarget.current_hp −= remaining_dmg
10Death Checkhp ≤ 0 → emit unit_died, disable, death FX
11Floating NumberDamageNumber3D at target pos (crit = yellow)

L.6 Runtime Bugs Found & Fixed During Godot Testing
#BUG / RESOLUTION
1modulate not declared on CharacterBody3D
Symptom: Runtime error — modulate property not found on BattleUnit (CharacterBody3D).
Cause: modulate is a CanvasItem property (2D only). BattleUnit extends CharacterBody3D (3D).
Fix: Replaced with GeometryInstance3D.transparency. Added recursive _set_mesh_transparency(node, value). Spawn protection tweens 0.7 → 0.0.
File: Scripts/Systems/BattleUnit.gd
2Invalid access to is_alive on previously freed instance
Symptom: Crash after first unit death — accessing .is_alive on a freed BattleUnit reference.
Cause: Death tween calls queue_free(), but player_units / enemy_units arrays still hold stale references.
Fix: Added is_instance_valid(u) guards at all 9 access points across BattleManager.gd (6) and BattleUnit.gd (3).
Files: Scripts/Managers/BattleManager.gd, Scripts/Systems/BattleUnit.gd
3filter() lambda type-cast crash on freed objects
Symptom: Victory filters return empty arrays — instant DRAW after first death.
Cause: Typed lambda func(u: BattleUnit) — GDScript casts argument before lambda body runs. Freed object fails cast before is_instance_valid() can help.
Fix: Removed type hints: func(u) instead of func(u: BattleUnit). Changed result arrays to untyped Array.
File: Scripts/Managers/BattleManager.gd (lines 366, 369)
4Integer division warning in timer display
Symptom: Godot console warning on int / int truncation.
Cause: int(battle_time) / 60 performs integer division, flagged by GDScript 4.x.
Fix: Changed to int(battle_time / 60.0) — float division first, then explicit truncation.
File: UI/Screens/BattleHUD.gd (line 51)

L.9.3 Generation Pipeline (13 Steps)
#STEPSPECIFICATION
1Seed InitializationAccept seed (user-provided or time-based). Init seeded PRNG (mulberry32). All randomness flows through this.
2System CountPick random count within size range (S:12–14, M:16–20, L:22–26, X:28–35). Fewer active factions = more neutral systems (count stays the same).
3Place Nexus + HomeworldsNexus at (510, 320). Place N faction homeworlds (N = active factions, 2–5) via spaced radial placement: ~360°/N apart ±jitter, radius 0.65–0.90 × galaxy_radius. Each gets a guaranteed Capital World. Enforce ≥3 lane-hop separation.
4Generate SystemsPlace remaining (total − 1 − N) systems via jittered radial placement on 1024×640 space. Min distance 80px. Inner ring (0.2–0.4r): White Dwarf/Blue bias. Mid (0.4–0.7r): Yellow/Binary. Outer (0.7–1.0r): Red Giant/any.
5Assign Star Types5 types: Yellow Main Seq, Red Giant, Blue Supergiant, Binary, White Dwarf. Distribution biased by ring. Fisher-Yates shuffle of type pool.
6Generate PlanetsCount by star type (Yellow:5-8, Red:6-10, Blue:4-7, Binary:3-6, WD:2-5). Planet type weighted by star. Each gets: type, territory count, base yield, traits.
7Generate Warp LanesFor each system: connect to 2 nearest, 50% chance for 3rd. Dedupe edges. Ensure 1–3 chokepoints (prune longest edges).
7bRoll Orbital FeaturesFor each planet, filter ORBITAL_FEATURES by valid planet types, weighted pick. Keys: asteroid_belt, ring_system, debris_field, none.
8Post-Lane PruningRemove direct Homeworld↔Nexus lane for ALL factions. BFS flood fill. If disconnected subgraph found, add minimum bridge lane. Verify all homeworlds ≥3 lane-hops apart (re-add/remove lanes if needed).
9Place Wormholes1–2 pairs connecting non-adjacent systems (>60% galaxy diameter apart). Collapse chance: 5–40% per use, fixed per seed.
10Faction PlacementN homeworlds already placed in Step 3. Nexus = Guardians. Vorax at 1–2 edge systems (not homeworlds). Remaining systems = neutral/unoccupied. Elimination: all planets lost + no fleets in transit = destroyed.
11Assign Planet TraitsRoll from Ch13 tables: Resource (Rich Veins 20% Volcanic, Flux 15% Ocean), Combat (Fortified 15% Capital, Low-G 100% Moon), Strategic (Precursor 30% Ruins, Guardian Echo 15% Ruins/WD). Max 1 per category per planet.
12Territory AdjacencyPer planet: build connected territory graph (DoW Soulstorm style). Ring backbone 1→2→…→N→1 guarantees all reachable. Add 1–3 seeded cross-links for strategic shortcuts. Max 5 connections per territory. Store as adjacency dict { 1: [2, N], … }.
12bRoll System PhenomenaPer-system weighted pick. nebula, ion_storm, gravity_well, radiation_field, solar_flare_zone, none. Homeworlds always none. Nexus always gravity_well.
13OutputReturn { seed, size, systems[], lanes[], wormholes[], homeworlds{} }. Each system: { id, name, position, star_type, planets[], faction, is_homeworld, lanes[] }. Each planet: { type, territory_count, territory_adjacency{}, base_yield, traits[] }.
L.9.4 Seeded RNG Requirement
Same seed MUST produce identical galaxy across all platforms.
Implementation: mulberry32 seeded PRNG. All random calls routed through it.
Fisher-Yates shuffle for all randomized array selections.
Godot's built-in randi() not used — not guaranteed deterministic cross-platform.

L.10.2 — TRAVEL TIME ORBITAL IMPACT
INTRA-SYSTEM
Base 1 cycle. −0.5 if planets within 60° (same side), +0.5 if >180° apart (opposite sides). Net: 0.5–1.5 cycles.
INTER-SYSTEM
current_distance / base_distance modifier (0.95–1.05). Displayed as Optimal / Standard / Extended.
LAUNCH WINDOWS
Open when two systems at closest approach. 15% travel discount, lasts 2–3 cycles, occurs every 30–50 cycles per lane. Shown as green pulse on warp lane.

L.14.2 — STAR TWINKLING
Sine wave modulation per star. Each star has a random phase offset and speed. Alpha oscillates between base_alpha × 0.5 and base_alpha × 1.0. Applied per-frame in _process(delta).
L.14.3 — NEBULA CLOUD PATCHES
2–4 soft nebula patches placed randomly across the map. Each patch is rendered as concentric circles with low alpha (0.01–0.03). Colors drawn from faction palette. Purely cosmetic — no gameplay effect.
L.14.4 — SEPARATE RNG
Visual layer uses a dedicated RNG with seed 42. This is completely separate from the galaxy generation RNG. Ensures that adding/removing visual elements never changes the procedurally generated galaxy layout.

L.14.6 — GALACTIC CORE GLOW
Concentric warm gradient centered on the Nexus system (512, 320). Multiple circles radiate outward with decreasing alpha, blending from warm gold/amber at the center to transparent at the edges. Creates a visual “galactic core” focal point that draws the eye to the Nexus objective.
L.14.7 — SPIRAL ARM STAR CLUSTERING
Logarithmic spiral density applied to background star placement. Stars cluster along spiral arm paths radiating from the galactic center, creating a more natural galaxy appearance. Uses the visual RNG (seed 42) — does not affect game system placement.
L.14.8 — STAR-TYPE NODE COLORS
Each system node’s fill color reflects its star type (yellow for yellow_main_seq, red for red_giant, blue for blue_supergiant, white for white_dwarf, dual-tone for binary). An outer ring color indicates faction ownership. Unowned systems show a neutral grey ring.
L.14.9 — LANE PULSE ANIMATION
Energy dots travel along warp lanes at a steady pace. Multiple dots per lane, evenly spaced, animated each frame via _process(delta). Dot positions wrap at lane endpoints. Gives the galaxy a living, connected feel without impacting gameplay logic.
L.14.10 — WORMHOLE PORTALS
Spinning purple spirals rendered at each wormhole endpoint. Drawn as arcs that rotate over time, creating a swirling portal effect. Both ends of a wormhole pair display matching portals. Wormhole connections are drawn as dashed lines between paired systems.
L.14.11 — HOMEWORLD BEACONS
Pulsing rings + compass tick marks surround each faction’s homeworld system. The ring expands and contracts on a sine cycle, drawing attention to the faction’s capital. Four compass ticks (N/S/E/W) extend outward from the ring. Colored in the faction’s primary color.
L.14.12 — SYSTEM SIZE SCALING
System node radius scales by planet count. Systems with more planets render as larger circles on the galaxy map, providing an at-a-glance indication of system value. Homeworlds receive an additional size bonus to ensure they are always prominent.
L.14.13 — FACTION INFLUENCE ZONES
Soft colored halos drawn around faction-owned systems. Each faction’s influence zone uses that faction’s primary color at very low alpha, creating a territorial “fog” that visually groups owned systems. Overlapping zones blend naturally. Drawn on layer 4, below lanes and nodes.
L.15 — Turn Phase Pipeline Architecture (Godot 4.4 Rewrite)
“Full 6-phase turn cycle with adaptive phase skipping and conflict-driven resolution.”
L.15.1 — 6-PHASE CYCLE
#PHASEAUTO-DELAYDESCRIPTION
1MAP—Player input phase: issue fleet orders, manage units, review territory
2ARMADA0.5sExecute fleet moves, run conflict detection (multi-faction fleet presence)
3APPROACHskippablePre-battle fleet positioning (skipped if no conflicts)
4SPACE0.3sSpace combat resolution (skipped if no conflicts)
5GROUND0.3sGround combat resolution (skipped if no conflicts)
6RESOLUTION—Collect resources via supply chain, apply combat results, advance turn counter
L.15.2 — ADAPTIVE PHASE SKIPPING
The pipeline skips APPROACH, SPACE, and GROUND phases when no conflicts are detected during the ARMADA phase. Conflict detection checks for multi-faction fleet presence at any system. If no conflicts exist, the pipeline jumps directly from ARMADA to RESOLUTION, keeping peaceful turns fast.
L.15.3 — MVP CONFLICT RESOLUTION
In the MVP implementation, conflict resolution is simplified: the attacker captures one territory per conflict. Full space and ground battle scenes will be wired in later phases. This allows the turn pipeline to function end-to-end while battle mechanics are developed independently.
L.15.4 — RESOURCE COLLECTION (RESOLUTION PHASE)
During RESOLUTION, each faction collects yields from all owned territories. Yields are modified by the supply chain hop distance from homeworld (see §15.9). Credits, scrap, influence, and intel are accumulated per-territory and added to faction totals. The turn counter then advances and the pipeline returns to MAP.
